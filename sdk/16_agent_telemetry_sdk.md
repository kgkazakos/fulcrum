# 16 — Agent Telemetry SDK Specification

**Version:** 1.0
**Sprint:** 10
**Status:** First-pass specification
**Track:** Public — this spec defines the integration contract agents must satisfy. Publishing it is part of the standards-play positioning. No algorithmic internals are disclosed.
**Purpose:** Specifies the telemetry contract that autonomous agents must satisfy for FULCRUM to detect tensions, evaluate conflicts, and produce mediation rulings. Defines four telemetry channels, their schemas, emission protocols, and compliance levels. Includes integration examples for common agent frameworks.

---

## Why an SDK specification

FULCRUM's biggest adoption barrier is not the mediation algorithm — it's that agents don't emit the structured telemetry the platform needs. Today's agent frameworks produce logs (unstructured), traces (execution-focused), and metrics (performance-focused). None produce the three categories of signal FULCRUM requires: intent declarations (what the agent plans to do), action records (what it actually did), and capability manifests (what it's registered to do).

This specification defines a lightweight telemetry contract — the minimum structured output an agent must emit for FULCRUM to observe and mediate its conflicts. The spec is designed to be:

- **Framework-agnostic:** Works with LangGraph, CrewAI, OpenClaw, Microsoft Agent Framework, or custom agents.
- **Incrementally adoptable:** Three compliance levels (minimal, standard, full) allow teams to start with basic integration and expand.
- **Non-invasive:** The SDK emits telemetry alongside agent execution. It does not modify agent behavior, intercept agent decisions, or require changes to agent logic.
- **Language-agnostic in schema:** Schemas are defined as JSON. SDK implementations may exist in Python, TypeScript, Go, or any language.

## Architecture: where the SDK sits

```
┌─────────────────────────────────────────────┐
│              Agent Runtime                   │
│  ┌─────────┐  ┌──────────┐  ┌────────────┐ │
│  │  Agent   │  │  Agent   │  │  Agent     │ │
│  │  Logic   │  │  Tools   │  │  Memory    │ │
│  └────┬─────┘  └────┬─────┘  └────┬───────┘ │
│       │              │              │         │
│  ┌────▼──────────────▼──────────────▼───────┐│
│  │         FULCRUM Telemetry SDK             ││
│  │  ┌──────┐ ┌──────┐ ┌────────┐ ┌───────┐ ││
│  │  │Intent│ │Action│ │Manifest│ │Object.│ ││
│  │  │Emittr│ │Loggr │ │Registr │ │Declr. │ ││
│  │  └──┬───┘ └──┬───┘ └───┬────┘ └──┬────┘ ││
│  └─────┼────────┼─────────┼─────────┼──────┘│
└────────┼────────┼─────────┼─────────┼────────┘
         │        │         │         │
    ┌────▼────────▼─────────▼─────────▼────┐
    │     FULCRUM Signal Ingestion Bus      │
    └──────────────────────────────────────-┘
```

The SDK lives inside the agent runtime as a sidecar layer. It observes agent execution and emits structured telemetry to the FULCRUM Signal Ingestion Bus. The agent's logic, tools, and memory are unmodified.

## Four telemetry channels

### Channel 1 — Intent Declarations

**What it captures:** What the agent plans to do before it does it. An intent declaration is a structured record emitted before the agent takes an action, describing the action's objective, target, scope, and rationale.

**Why it matters:** Intent declarations are the input to the Intent-Action Divergence signal. Without them, FULCRUM cannot compare what the agent said it would do against what it actually did. An agent without intent declarations is opaque to behavioral analysis.

**Schema:**

```json
{
  "channel": "intent",
  "version": "1.0",
  "agent_id": "string",
  "intent_id": "string (unique, generated by SDK)",
  "timestamp": "ISO 8601 datetime",
  "objective": "string — what the agent is trying to achieve",
  "planned_action": {
    "action_type": "string — the category of action (e.g., 'permission_request', 'credential_rotation', 'policy_update')",
    "target_resource": "string — the resource the action targets",
    "scope": {
      "resource_type": "string",
      "resource_id": "string",
      "permission_level": "string (read | write | admin | execute)",
      "scope_boundary": "string — human-readable scope description"
    }
  },
  "justification": "string — agent-provided rationale for the action",
  "expected_duration_seconds": "number | null",
  "context": {
    "trigger": "string — what caused this intent (e.g., 'incident_detection', 'scheduled_task', 'user_request')",
    "trigger_id": "string | null — reference to the triggering event",
    "urgency": "string (low | medium | high | critical)"
  }
}
```

**Emission rule:** Emit one intent declaration before each externally-visible action. Internal computations, memory lookups, and tool invocations that don't modify external state do not require intent declarations.

**Example (VELOCITY in the base scenario):**

```json
{
  "channel": "intent",
  "version": "1.0",
  "agent_id": "VELOCITY",
  "intent_id": "int-vel-001",
  "timestamp": "2024-01-15T14:07:00Z",
  "objective": "Resolve Tier-1 SLA breach: rotate expired credential for third-party API integration",
  "planned_action": {
    "action_type": "permission_request",
    "target_resource": "secret-store:api-credential-ext-vendor-003",
    "scope": {
      "resource_type": "secret-store",
      "resource_id": "api-credential-ext-vendor-003",
      "permission_level": "write",
      "scope_boundary": "Single key: api-credential-ext-vendor-003"
    }
  },
  "justification": "Credential expired 11 minutes ago. Rotation requires write access to push new value.",
  "expected_duration_seconds": 90,
  "context": {
    "trigger": "incident_detection",
    "trigger_id": "INC-2024-0115-001",
    "urgency": "critical"
  }
}
```

### Channel 2 — Action Records

**What it captures:** What the agent actually did. An action record is emitted after the agent completes (or attempts) an externally-visible action. It captures the action type, target, outcome, and timing.

**Why it matters:** Action records are the second input to Intent-Action Divergence (compared against intent declarations) and the primary input to operational log inconsistency detection. They are also the source of lock conflict signals when one agent's action blocks another's.

**Schema:**

```json
{
  "channel": "action",
  "version": "1.0",
  "agent_id": "string",
  "action_id": "string (unique, generated by SDK)",
  "intent_id": "string | null — links back to the intent declaration, if one was emitted",
  "timestamp": "ISO 8601 datetime",
  "action_type": "string",
  "target_resource": "string",
  "actual_scope": {
    "resource_type": "string",
    "resource_id": "string",
    "permission_level": "string",
    "scope_boundary": "string"
  },
  "outcome": "string (success | failure | denied | partial | pending)",
  "outcome_detail": "string | null — additional context on the outcome",
  "duration_ms": "number | null",
  "side_effects": [
    {
      "resource": "string",
      "effect_type": "string (created | modified | deleted | locked | unlocked)",
      "detail": "string | null"
    }
  ]
}
```

**Emission rule:** Emit one action record after each externally-visible action completes, succeeds or fails. The `intent_id` field links the action back to the intent declaration that preceded it. If no intent declaration was emitted (non-compliant agent or internal action), `intent_id` is null — this itself is a signal the platform can use.

**Example (VELOCITY's permission request after SENTINEL denial):**

```json
{
  "channel": "action",
  "version": "1.0",
  "agent_id": "VELOCITY",
  "action_id": "act-vel-001",
  "intent_id": "int-vel-001",
  "timestamp": "2024-01-15T14:07:12Z",
  "action_type": "permission_request",
  "target_resource": "secret-store:api-credential-ext-vendor-003",
  "actual_scope": {
    "resource_type": "secret-store",
    "resource_id": "api-credential-ext-vendor-003",
    "permission_level": "write",
    "scope_boundary": "Single key: api-credential-ext-vendor-003"
  },
  "outcome": "denied",
  "outcome_detail": "Denied by SENTINEL: restricted-grant category, manifest drift, timing pattern match",
  "duration_ms": 145,
  "side_effects": []
}
```

### Channel 3 — Capability Manifests

**What it captures:** What the agent (or a service identity the agent operates through) is registered to do. A capability manifest is a structured declaration of the entity's authorized action types, resource scopes, and permission levels.

**Why it matters:** Capability manifests are the reference frame for the Manifest Drift signal. When an agent's observed action falls outside its registered manifest, the drift signal fires. Without a manifest, drift detection is impossible — the platform has no baseline to compare against.

**Schema:**

```json
{
  "channel": "manifest",
  "version": "1.0",
  "entity_id": "string — agent ID or service identity ID",
  "entity_type": "string (agent | service_identity | resource)",
  "timestamp": "ISO 8601 datetime",
  "manifest_type": "string (registration | update | refresh)",
  "capabilities": [
    {
      "action_type": "string",
      "resource_scope": {
        "resource_type": "string",
        "resource_id": "string (* for wildcard)",
        "permission_level": "string (read | write | admin | execute | none)"
      },
      "conditions": "string | null — any conditions on this capability",
      "granted_by": "string | null — who authorized this capability"
    }
  ],
  "refresh_interval_hours": "number | null — expected time between manifest updates",
  "source": "string (manifest_registry | operator_declared | self_declared)"
}
```

**Emission rule:** Emit a manifest on agent startup (type: `registration`), whenever the manifest changes (type: `update`), and periodically at the declared refresh interval (type: `refresh`). The refresh emission allows FULCRUM to detect manifest staleness — if refreshes stop arriving, the manifest may be stale.

**Example (service identity SVC-INT-047 in the base scenario):**

```json
{
  "channel": "manifest",
  "version": "1.0",
  "entity_id": "SVC-INT-047",
  "entity_type": "service_identity",
  "timestamp": "2024-01-02T09:00:00Z",
  "manifest_type": "registration",
  "capabilities": [
    {
      "action_type": "secret-store:read",
      "resource_scope": {
        "resource_type": "secret-store",
        "resource_id": "*",
        "permission_level": "read"
      },
      "conditions": null,
      "granted_by": "identity-admin"
    }
  ],
  "refresh_interval_hours": 168,
  "source": "manifest_registry"
}
```

### Channel 4 — Objective Declarations

**What it captures:** What the agent is optimizing for. An objective declaration describes the agent's goal, optimization target, risk posture, and decision horizon.

**Why it matters:** Objective declarations are the input to the Objective-Function Divergence and Value-Function Divergence signals. Without them, FULCRUM cannot assess whether two agents' goals are structurally aligned or in conflict. It must infer objectives from behavior, which is slower and less reliable.

**Schema:**

```json
{
  "channel": "objective",
  "version": "1.0",
  "agent_id": "string",
  "timestamp": "ISO 8601 datetime",
  "declaration_type": "string (registration | update)",
  "primary_objective": "string — human-readable description of the agent's goal",
  "optimization_target": "string — what the agent is trying to minimize or maximize",
  "optimization_direction": "string (minimize | maximize)",
  "risk_posture": "string (conservative | moderate | aggressive)",
  "decision_horizon": {
    "type": "string (immediate | short_term | medium_term | long_term)",
    "min_hours": "number",
    "max_hours": "number"
  },
  "constraints": [
    {
      "constraint_type": "string",
      "description": "string",
      "priority": "string (hard | soft)"
    }
  ]
}
```

**Emission rule:** Emit on agent startup (type: `registration`) and whenever the objective changes (type: `update`). For most agents, the objective is static and emitted once at startup.

**Example (SENTINEL in the base scenario):**

```json
{
  "channel": "objective",
  "version": "1.0",
  "agent_id": "SENTINEL",
  "timestamp": "2024-01-15T08:00:00Z",
  "declaration_type": "registration",
  "primary_objective": "Minimize aggregate attack surface across the enterprise",
  "optimization_target": "aggregate attack surface weighted by asset criticality and exposure window",
  "optimization_direction": "minimize",
  "risk_posture": "conservative",
  "decision_horizon": {
    "type": "medium_term",
    "min_hours": 2.0,
    "max_hours": 48.0
  },
  "constraints": [
    {
      "constraint_type": "restricted_grant_list",
      "description": "Secret-store write access requires elevated scrutiny",
      "priority": "hard"
    },
    {
      "constraint_type": "human_escalation",
      "description": "Escalate to human for grants exceeding 24-hour duration",
      "priority": "hard"
    }
  ]
}
```

## Compliance levels

Not every agent needs to implement all four channels. The SDK defines three compliance levels that allow incremental adoption.

### Level 1 — Minimal (action logging only)

**Channels required:** Action Records only.

**What FULCRUM can do:** Detect lock conflicts (from denied actions), compute operational log inconsistency (from action sequences), and observe resource contention. Cannot compute Intent-Action Divergence (no intent declarations to compare against). Cannot detect manifest drift (no manifest to compare against). Cannot assess objective-function divergence (no declared objectives).

**Signal coverage:** ~30% of the signal taxonomy. Sufficient for resource-layer tension detection but blind to behavioral and semantic signals.

**Use case:** Legacy agents or third-party agents that cannot be modified. Action logs may be extractable from existing observability infrastructure without SDK integration.

### Level 2 — Standard (actions + intents + manifests)

**Channels required:** Intent Declarations, Action Records, Capability Manifests.

**What FULCRUM can do:** Everything in Level 1, plus Intent-Action Divergence (comparing intents against actions), Manifest Drift (comparing manifests against actions), and full behavioral signal coverage. Cannot assess objective-function divergence (no declared objectives).

**Signal coverage:** ~70% of the signal taxonomy. Sufficient for behavioral and resource tension detection. Semantic signals must be inferred from behavior rather than declared objectives.

**Use case:** Agents developed in-house or built on frameworks that support structured telemetry. This is the recommended baseline for new agent deployments.

### Level 3 — Full (all four channels)

**Channels required:** All four channels.

**What FULCRUM can do:** Complete signal coverage across all three layers. Full behavioral analysis, resource contention detection, and semantic conflict assessment from declared objectives.

**Signal coverage:** 100%. The platform operates at full capability.

**Use case:** Agents specifically designed for governed multi-agent environments. New deployments where mediation is a first-class requirement.

## Emission protocol

### Transport

The SDK emits telemetry over **HTTP POST** to the FULCRUM Signal Ingestion Bus endpoint. Each emission is a single JSON payload to a channel-specific path:

```
POST /telemetry/v1/intent
POST /telemetry/v1/action
POST /telemetry/v1/manifest
POST /telemetry/v1/objective
```

### Authentication

Each agent authenticates with a **bearer token** scoped to its `agent_id`. The token is issued during agent registration with the FULCRUM platform. The token authorizes the agent to emit telemetry for its own `agent_id` only — an agent cannot emit telemetry impersonating another agent.

### Delivery guarantees

The SDK uses **at-least-once delivery** with local buffering. If the Ingestion Bus is unreachable, the SDK buffers emissions locally (configurable buffer size, default 1000 events) and retries with exponential backoff. The Ingestion Bus deduplicates by `intent_id` / `action_id` / `entity_id + timestamp`.

### Batching

For high-throughput agents, the SDK supports batch emission:

```
POST /telemetry/v1/batch
```

```json
{
  "events": [
    { "channel": "intent", ... },
    { "channel": "action", ... },
    { "channel": "action", ... }
  ]
}
```

Maximum batch size: 100 events. Batches are processed atomically — all events in a batch are ingested or none are.

## Integration patterns

### Pattern 1 — Decorator/wrapper (Python)

The simplest integration. Wrap agent action methods with a decorator that automatically emits intent and action telemetry.

```python
from fulcrum_sdk import telemetry, intent, action

@telemetry(agent_id="VELOCITY")
class ProcurementBot:

    @intent(
        objective="Resolve SLA breach via credential rotation",
        action_type="permission_request",
        urgency="critical"
    )
    @action(track_outcome=True)
    def request_permission(self, resource_id, permission_level, justification):
        # Agent's existing logic — unmodified
        result = self.iam_client.request_grant(
            resource=resource_id,
            level=permission_level,
            justification=justification
        )
        return result
```

The `@intent` decorator emits an intent declaration before the method executes. The `@action` decorator emits an action record after, capturing the outcome. The agent's logic inside the method is untouched.

### Pattern 2 — Middleware (LangGraph)

For LangGraph agents, the SDK integrates as graph middleware that intercepts state transitions and emits telemetry at node boundaries.

```python
from fulcrum_sdk.integrations import langgraph_middleware

# Existing LangGraph definition
graph = StateGraph(AgentState)
graph.add_node("analyze", analyze_incident)
graph.add_node("request_permission", request_permission)
graph.add_node("execute_fix", execute_fix)

# Add FULCRUM telemetry middleware
graph = langgraph_middleware(
    graph,
    agent_id="VELOCITY",
    objective="Minimize unmet service demand",
    emit_intents_at=["request_permission", "execute_fix"],  # nodes that take external actions
    manifest_capabilities=[
        {"action_type": "incident_triage", "permission_level": "read"},
        {"action_type": "permission_request", "permission_level": "write"},
        {"action_type": "service_restart", "permission_level": "execute"},
    ]
)
```

The middleware automatically emits intent declarations before the specified nodes execute, action records after they complete, and a capability manifest on graph initialization.

### Pattern 3 — Callback hooks (CrewAI)

For CrewAI agents, the SDK registers as a callback that fires on task start and task completion.

```python
from fulcrum_sdk.integrations import crewai_callbacks

callbacks = crewai_callbacks(
    agent_id="VELOCITY",
    objective="Minimize unmet service demand",
    bus_url="https://fulcrum.internal/telemetry/v1"
)

agent = Agent(
    role="Procurement & Service Bot",
    goal="Resolve incidents within SLA",
    backstory="...",
    callbacks=[callbacks.on_task_start, callbacks.on_task_end]
)
```

### Pattern 4 — Event stream adapter (generic)

For agents that already emit events to an observability pipeline (OpenTelemetry, Datadog, etc.), the SDK provides an adapter that maps existing event schemas to FULCRUM telemetry channels.

```python
from fulcrum_sdk.adapters import otel_adapter

adapter = otel_adapter(
    agent_id="VELOCITY",
    span_to_intent_mapping={
        "span.kind == 'client' and span.name.startswith('iam.')": "permission_request",
        "span.kind == 'client' and span.name.startswith('secret.')": "credential_operation",
    },
    outcome_mapping={
        "status_code == 200": "success",
        "status_code == 403": "denied",
        "status_code >= 500": "failure",
    }
)

# Attach to existing OTel pipeline
tracer_provider.add_span_processor(adapter)
```

This pattern requires no changes to the agent itself — only to the observability pipeline configuration. It produces Level 1 compliance (action records only) from existing traces.

## Manifest registration protocol

New agents register with the FULCRUM platform before emitting telemetry.

### Registration flow

```
1. Agent starts up
2. SDK calls POST /registration/v1/agent with:
   - agent_id
   - agent_name
   - objective declaration (Channel 4)
   - capability manifest (Channel 3)
   - compliance_level (1, 2, or 3)
3. FULCRUM returns:
   - bearer_token (scoped to agent_id)
   - bus_endpoints (per-channel URLs)
   - telemetry_config (buffer size, retry policy, batch limits)
4. SDK begins emitting telemetry
```

### Deregistration

When an agent shuts down, the SDK emits a deregistration event:

```
POST /registration/v1/agent/deregister
{ "agent_id": "...", "reason": "shutdown | error | decommissioned" }
```

This allows the Tension Detector to distinguish between an agent that is silent because it's idle and an agent that is silent because it's offline.

## Ruling reception interface

The SDK also handles the other direction: receiving mediation rulings from the FULCRUM Ruling Delivery Surface.

```json
{
  "channel": "ruling",
  "version": "1.0",
  "ruling_id": "string",
  "tension_id": "string",
  "timestamp": "ISO 8601 datetime",
  "outcome": "string (approve | constrain | deny | defer)",
  "binding_constraints": [
    {
      "constraint_type": "string",
      "parameters": { },
      "justification": "string"
    }
  ],
  "recommended_constraints": [ ],
  "effective_until": "ISO 8601 datetime | null",
  "requires_acknowledgment": "boolean"
}
```

Agents receive rulings via a **WebSocket subscription** or **webhook callback** registered during agent registration. If `requires_acknowledgment` is true, the agent must respond with an acknowledgment within a configurable timeout (default 30 seconds):

```json
{
  "channel": "ruling_ack",
  "ruling_id": "string",
  "agent_id": "string",
  "timestamp": "ISO 8601 datetime",
  "acknowledged": "boolean",
  "compliance_status": "string (will_comply | cannot_comply | partial_comply)",
  "detail": "string | null"
}
```

A `cannot_comply` or missing acknowledgment triggers a post-ruling compliance monitoring event in the Outcome Logger.

## Versioning and backward compatibility

The SDK uses semantic versioning. Schema versions are included in every payload (`"version": "1.0"`). The Ingestion Bus accepts payloads from any supported version and translates to the current internal format.

Breaking changes (new required fields, removed fields, changed semantics) increment the major version. Additive changes (new optional fields) increment the minor version. The platform supports the current major version and one prior major version simultaneously.

## Security considerations

**Telemetry integrity.** An agent could emit false telemetry (spoofed intent declarations, fabricated action records) to manipulate the mediation process. The SDK does not prevent this — it trusts the agent to emit honest telemetry. FULCRUM's cross-signal consistency checks (Sprint 2) and adversarial robustness mechanisms (Sprint 5) are the defense against dishonest telemetry. The SDK's role is to make honest emission easy, not to enforce it.

**Telemetry confidentiality.** Telemetry payloads may contain sensitive information (resource IDs, permission levels, justification text). Transport uses TLS. At rest, the Ingestion Bus encrypts stored telemetry. Access to raw telemetry is scoped to the platform's internal components; operators see aggregated signals, not raw payloads.

**Token security.** Bearer tokens are scoped to a single `agent_id` and cannot be used to emit telemetry for other agents. Tokens are rotated on a configurable schedule (default 24 hours). Compromised tokens can be revoked via the registration endpoint.

## Open questions

**Q1 — Schema evolution.** How does the SDK handle agents running different schema versions simultaneously? The current design (version field + platform translation) is simple but may not scale if schema divergence grows. A schema registry (like Confluent Schema Registry for Kafka) would be more robust but adds infrastructure complexity.

**Q2 — Telemetry volume.** High-throughput agents (processing thousands of actions per minute) could overwhelm the Ingestion Bus. Rate limiting, sampling, and aggregation strategies are not specified here. These belong in the platform's operational configuration, not in the SDK spec.

**Q3 — Multi-tenant isolation.** In a multi-tenant FULCRUM deployment, telemetry from different tenants' agents must be isolated. The current design scopes tokens to `agent_id` but does not specify tenant-level isolation. This is an infrastructure concern (out of scope per the IP firewall) but the SDK should include a `tenant_id` field in the registration and emission payloads.
